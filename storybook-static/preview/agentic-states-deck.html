Agentic interaction states

Six live-surface primitives that govern how the agent shows its work — what it's doing, what it's about to do, and where the human still has the wheel. Every state pairs with an event from the AG-UI protocol, every state surfaces user agency in-place.

Primitive · 1 of 6 02 / 08

AgentStatusBar

A single-row "what is the agent doing right now" header. Pinned to the top of any agentic surface — chat modal, run page, inbox detail. Reads activity, exposes session-level controls, and never disappears mid-run.

Working through plan · 3 of 6 · 14.2s · 2 tools · 1 handoff
Paused · awaiting your tie-break · step 4 of 6 · 22.0s elapsed
Done · 6 steps in 38s · 2 tools · 1 handoff · 0 errors
Rules
  • Always present, never hidden. Use to give every run a visible, cancellable identity.
  • Single moving element — the dot. No spinners, no progress bars; activity is signalled by the pulse.
  • Inject correction opens a one-line text input that becomes a user_message event the agent reads on its next turn.
  • Branch forks the run from this step; the audit trail diverges, both branches are kept.
  • Three states share the same chrome: thinking/streaming, paused (amber), done (green). State changes never re-flow surrounding content.

Primitive · 2 of 6 03 / 08

ToolCallCard

Inline card that surfaces a tool invocation between chat messages. Three states — about to run (gate), running, returned — with editable args and an approval ladder. Destructive tools always gate; read-only tools gate only when step-through mode is on.

About to call graph.write
step 4 · proposes 22 owner assignments · destructive · review args
Awaiting approval
action: "assign_owner" scope: "payments-svc" policy: "deploy_recency" batch_size: 22 notify: true
retries · 0 of 3 used
Calling graph.expand
depth 2 · 22 root nodes · 1.4s elapsed
Running
Returned cmdb.query · 22 rows
230ms · 22 orphaned CIs in payments-svc
OK
Rules
  • The tool name is always rendered in mono (graph.write). The card's identity is the tool, not a fuzzy paraphrase.
  • Editable args use a dashed underline + spruce 6% tint. Click to edit before approving — no separate dialog.
  • Destructive tools (graph.write, cmdb.write, incident.create) always gate, regardless of step-through mode.
  • Returned cards collapse by default. Expand to see args + result envelope; the run's audit log never loses fidelity.
  • Failure swaps the icon to warm-red and surfaces Retry / Skip / Branch. Never auto-retries silently.

Primitive · 3 of 6 04 / 08

HumanInputRequest

The agent has hit ambiguity it can't (or shouldn't) resolve alone. Renders inline in the message stream, on an amber surface that intentionally breaks the AI gradient — your eye has to find it on a long scroll. The run is paused until you answer.

Reconciliation Agent · needs your call 14:08:22

Two of the 22 CIs have ambiguous ownership. Pick a tie-break before I write.

web-gateway-02 and checkout-api-canary were last touched by A. Ortiz (deploy, structural) and K. Patel (config, cosmetic). Recency rule favours K. Patel; structural-edit rule favours A. Ortiz.
blocking · agent paused
Rules
  • The only primitive that breaks the AI gradient — by design. Amber + warm-red on a long scroll cuts through everything else.
  • Always blocking. The AgentStatusBar flips to paused the moment this renders.
  • Action buttons must be named choices, never "Yes/No" or "Continue/Cancel". The user picks a rule, the agent applies it.
  • Context block restates the priors that led here, in the user's vocabulary. Never echo the raw tool envelope.
  • Answering produces an input.received event; the run resumes from the same step, not the start.

Primitive · 4 of 6 05 / 08

StepTimeline

The agent's plan, rendered. Each step is a row with a marker, a label, a timestamp, and — on the active row — inline approval controls. Lives in the right rail of any agentic surface; in narrow surfaces, collapses to a horizontal stepper.

Rules
  • Active marker pulses spruce; done is solid green; failed is solid warm-red with an ✕. Never use color alone — every state pairs with a glyph.
  • The active step reveals row-level controls — Edit · Approve once · Approve all. Other rows are read-only.
  • Approve all disengages step-through for the rest of the run; AgentStatusBar flips to thinking/streaming with no further gates.
  • The plan is mutable up to the active step. Edit a pending step and the audit trail records both the original and the override.
  • Branching from any step duplicates the plan from that point and flips both runs into a side-by-side comparison view.

Primitive · 5 of 6 06 / 08

HandoffCard

Marks the moment one agent passes work to another agent or to a human. Inline in chat, between messages. Always two-sided — you see who's leaving, who's arriving, why, and what context travels with them.

Rules
  • Always two-sided. Left = origin, right = recipient. Both nodes show identity (avatar) and role (function).
  • The reason chip is required and must name a capability — "needs SQL skill", "review needed", "out-of-policy escalation".
  • The carrying ctx line lists the priors travelling with the handoff. Click to inspect the full state envelope.
  • Agent → Human handoffs flip the AgentStatusBar to waiting; Agent → Agent handoffs keep it thinking.
  • The arrow is a single vectorized stroke — same weight on every surface, never decorative.

Primitive · 6 of 6 07 / 08

StateDeltaToast

Ambient notification when the agent mutates a value visible elsewhere on the page — a trust score, a CI count, an owner name. Bottom-left, non-blocking, ephemeral. The audit trail is permanent; the toast is just the announce.

ambient stack · bottom-left
Agent updated 20 CI owners
unassigned→resolved · 2 still ambiguous
Trust score recomputed
62%→58% · payments domain
Inbox: 3 new items learned from this run
added 3 owner overrides
Rules
  • Bottom-left, never top-right. Right side is reserved for system-level OS toasts; left side is the agent's voice.
  • Stack vertically up to 3. New deltas push older ones up; older ones fade after 8s. Never auto-dismiss the most recent.
  • Always include a diff line — old value, arrow, new value. If there's no diff, it's not a state change; don't toast.
  • Undo is offered for any reversible delta. Why? opens the audit trail at this exact event.
  • Toasts are ephemeral; the source-of-truth is the inbox. Never put critical info only here.
